Time Frame Parameters

Optimization and Forecast Horizons

The Optimization Horizon (parameter 1) and the Forecast Horizon Week (parameter 44) work together to specify the number of weeks that pricing and forecasts will be produced.

You may want to increase parameter 1 in cases where residents give notice more than 16 weeks in advance. Keep in mind that the default (16) is the optimal horizon.

The Forecast Horizon (parameter 44) = longest lease term + edge weeks. By default, this is 52 + 4 = 56. You would change this if you offer longer lease terms to make sure that the system is forecasting far enough into the future.

1 Optimization Horizon

44 Forecast Horizon

Renewal Pricing Time Frames

The Renewal Window parameters set the time frame to be considered when setting renewal rates. These parameters are typically set once and not changed.

Parameter 146 specifies the number of weeks (or other unit of time as specified by parameter 145) for which the system produces renewal pricing. This should be less than or equal to the time specified in parameter 1 (Optimization Horizon).

You might change parameter 146 if you want to see renewals farther into the future. Be aware that extending beyond 16 weeks can impact statistical accuracy. You should not extend beyond 20 weeks without first contacting Support to review the forecast data.

Parameter 147 addresses how far in advance residents are contacted to discuss renewal options. A typical time period is 60 days.

Inferring a MTM Lease

Parameter 16 (MTM Lag Days) and parameter 17 (Number of days to give notice) are used to tell the system when to infer a MTM lease for a resident that has not renewed or given notice.

For example, if parameter 17 is 60 and parameter 16 is 45, the system will infer a MTM lease for any undecided expiring leases 15 days (60-45) prior to the lease end.

16 MTM Lag Days

17 Number of days to give notice

Seasonality

Parameter 54 specifies the number of months of historical data that the system should use when calculating seasonality. Using at least 18 months of historical data is recommended. The minimum is 13 months. You would reduce the number of months when the accuracy of the data is in question. You would increase the number of months when quality data is available for more months. In most cases, the default (36 months) is used.

54 Historical Days Limit (months)

Leasing Velocity

Parameter 88 tells the system the number of days worth of observations to use when calculating leasing velocity. Leasing velocity is calculated in 7 day increments with the value rounded up to the next highest multiple. For example:

30 days/7 = 5 weeks

27 days/7 = 4 weeks

You might change this parameter due to seasonal patterns. For example, during a peak season, fewer weeks could be factored into leasing velocity for more reactivity.

Parameter 104 determines the weight that the system assigns to each week of guest card activity when calculating leasing velocity. A lower number gives greater influence to recent weeks.
